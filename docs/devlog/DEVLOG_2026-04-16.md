# Devlog — 2026-04-16 — Gestión Cementerio Somahoz

Esta sección resume **lo trabajado el 16-04** para arrancar el módulo de **Gestión del Cementerio Municipal (Somahoz)** siguiendo el stack y convenciones de Conect@ 2.0.

## Contexto y objetivo

- **Objetivo**: arrancar desde cero el módulo de Cementerio (local), dejando listo:
  - esquema de BD (migraciones) **agnóstico SQLite/MySQL**
  - modelos Eloquent con relaciones

  - UI target del ecosistema: PrimeVue 4 + preset Aura (pendiente de integrar visualmente de forma completa)

## Estado del repo hoy (qué hay implementado)

### 1) Base de datos (migraciones)

Se crearon migraciones para replicar el **schema de referencia** del SQL de producción (`docs/sql/basedatosnueva.sql`) de forma compatible con SQLite y MySQL.

Migraciones creadas en `database/migrations/`:
- `cemn_cementerios`
- `cemn_zonas` (FK a cementerio)
- `cemn_bloques` (FK a zona; `filas`/`columnas` + `tipo`)
- `cemn_sepulturas` (celdas; FK a zona y opcional a bloque; `fila`/`columna`/`estado`)
- `cemn_terceros` (personas)
- `cemn_concesiones` (FK a sepultura + auto-referencia `concesion_previa_id`)
- `cemn_concesion_terceros` (pivot concesión-tercero con `rol`)
- `cemn_difuntos` (FK opcional a tercero y a sepultura)
- `cemn_documentos` (FK a sepultura)
- `cemn_fuentes`
- `cemn_movimientos` (FK a difunto y sepulturas origen/destino)
- `cemn_registro_fuentes` (FK a fuente)

Notas importantes:
- En el SQL real la “unidad” individual es `cemn_sepulturas` (no `cemn_unidades`).
- Se evitó usar tipos exclusivos de SQLite (objetivo: migraciones válidas también para MySQL).

### 2) Modelos Eloquent

- `CemnSepultura` → `belongsTo` zona y bloque; `hasMany` concesiones/difuntos/documentos; helpers de estado + `generarCodigo()`
- `CemnTercero` → `belongsToMany` concesiones vía pivot; accessor `nombre_completo`
- `CemnConcesion` → `belongsTo` sepultura; `belongsToMany` terceros vía pivot; relación concesionario activo
- `CemnConcesionTercero` (pivot como modelo)
- `CemnDifunto` → `belongsTo` sepultura; `hasMany` movimientos
- `CemnDocumento`
- `CemnFuente`
- `CemnMovimiento`
- `CemnRegistroFuente`

## Problemas reales encontrados hoy (y cómo se resolvieron)

1) **PowerShell ExecutionPolicy** bloqueaba scripts (`PSSecurityException`).
   - Solución: ejecutar con:
     - `powershell -ExecutionPolicy Bypass -File .\scripts\...ps1`

2) Los scripts se estaban ejecutando desde `C:\Windows\system32`.
   - Solución: `cd` al directorio del repo antes de ejecutar.

3) Composer no estaba disponible.
   - Solución: `full-setup.ps1` descarga `tools/composer.phar`.

4) Error Composer: **OpenSSL extension missing**.
   - Significado: el PHP en uso no tenía `openssl` habilitado; Composer no puede descargar por HTTPS.
   - Solución: habilitar OpenSSL en `php.ini` (Laragon) y reiniciar.

5) Error `Could not open input file: artisan`.
   - Significado: faltaba skeleton completo de Laravel.
   - Solución: `full-setup.ps1` genera Laravel 12 automáticamente si falta `artisan`.

6) Conflicto npm `ERESOLVE` (ApexCharts).
   - Causa: `vue3-apexcharts` requiere `apexcharts >= 5.10.0`.
   - Solución: se actualizó `package.json` para ser compatible.

## Problemas reales de arranque (y fixes)

3) **Vite apuntaba a `0.0.0.0`** en el HTML y dejaba la SPA en blanco.

4) **Vue warning (runtime compilation no soportada)** por usar `template` en el root.

5) **Vite manifest not found** (`public/build/manifest.json`) en dev.
   - Fix: levantar Vite (contenedor `node`) y asegurar `public/hot`/dev server operativo.

6) **Permisos / cache de Spatie fallaban por permisos de filesystem**.
   - Síntoma: `file_put_contents(...storage/framework/cache/...): Permission denied`
   - Fix:
     - permisos amplios en `storage/` y `bootstrap/cache/` dentro de contenedor
     - `.env` → `CACHE_STORE=array` para desarrollo (evita escrituras en disco)

## 4) Operativa real: “no se actualiza la página” (Vite hot vs build)

Se observó que a veces el navegador seguía mostrando “Dashboard”/estilos antiguos aunque el código local estuviera cambiado.

Puntos clave:
- Si existe `public/hot`, Laravel carga assets desde Vite dev server (`5173`).
- Si NO existe, carga assets versionados desde `public/build` (manifest).
- El navegador puede cachear HTML/JS de manera agresiva en dev; el build con hashes reduce problemas, pero el HTML puede quedarse en caché en el browser.

Acciones usadas para forzar consistencia (sobre todo en Docker):
- Reiniciar Vite (`docker compose --profile vite restart node`) si se usa HMR.
- Si hay dudas, **desactivar hot**:
  - borrar `public/hot`
  - ejecutar `npm run build` en el contenedor `node`
  - reiniciar `nginx` + `app`
- Limpiar cachés Laravel:
  - `php artisan optimize:clear`

Resultado recomendado para demos estables:
- Modo “build” (sin `public/hot`) y con `public/build` generado.

**Fix**:
