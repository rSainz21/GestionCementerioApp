## Estado del repo hoy (qué hay implementado)

### 1) Base de datos (migraciones)

Se crearon migraciones para replicar el **schema de referencia** del SQL de producción (`docs/sql/basedatosnueva.sql`) de forma compatible con SQLite y MySQL.

Migraciones creadas en `database/migrations/`:
- `cemn_cementerios`
- `cemn_zonas` (FK a cementerio)
- `cemn_bloques` (FK a zona; `filas`/`columnas` + `tipo`)
- `cemn_sepulturas` (celdas; FK a zona y opcional a bloque; `fila`/`columna`/`estado`)
- `cemn_terceros` (personas)
- `cemn_concesiones` (FK a sepultura + auto-referencia `concesion_previa_id`)
- `cemn_concesion_terceros` (pivot concesión-tercero con `rol`)
- `cemn_difuntos` (FK opcional a tercero y a sepultura)
- `cemn_documentos` (FK a sepultura)
- `cemn_fuentes`
- `cemn_movimientos` (FK a difunto y sepulturas origen/destino)
- `cemn_registro_fuentes` (FK a fuente)

Notas importantes:
- En el SQL real la “unidad” individual es `cemn_sepulturas` (no `cemn_unidades`).
- Se evitó usar tipos exclusivos de SQLite (objetivo: migraciones válidas también para MySQL).

## Problemas reales encontrados hoy (y cómo se resolvieron)

1) **PowerShell ExecutionPolicy** bloqueaba scripts (`PSSecurityException`).
   - Solución: ejecutar con:
     - `powershell -ExecutionPolicy Bypass -File .\scripts\...ps1`

2) Los scripts se estaban ejecutando desde `C:\Windows\system32`.
   - Solución: `cd` al directorio del repo antes de ejecutar.

3) Composer no estaba disponible.
   - Solución: `full-setup.ps1` descarga `tools/composer.phar`.

4) Error Composer: **OpenSSL extension missing**.
   - Significado: el PHP en uso no tenía `openssl` habilitado; Composer no puede descargar por HTTPS.
   - Solución: habilitar OpenSSL en `php.ini` (Laragon) y reiniciar.

5) Error `Could not open input file: artisan`.
   - Significado: faltaba skeleton completo de Laravel.
   - Solución: `full-setup.ps1` genera Laravel 12 automáticamente si falta `artisan`.

6) Conflicto npm `ERESOLVE` (ApexCharts).
   - Causa: `vue3-apexcharts` requiere `apexcharts >= 5.10.0`.
   - Solución: se actualizó `package.json` para ser compatible.

## 2) Flujo restos / exhumación — rediseño completo

### Problema resuelto

La gestión de nichos con múltiples ocupantes (histórica o nueva) requería una lógica clara de estados.

### Lógica implementada

**Estado `estado_inhumacion` en `cemn_difuntos`:**
- `'inhumado'` — persona activa en el nicho
- `'restos'` — permanece físicamente en el nicho (`sepultura_id` conservado); nicho queda `libre` para nuevas inhumaciones
- `'exhumado'` — sale del nicho (`sepultura_id = null`); solo aparece en historial de movimientos

**Reglas de negocio:**
- Al **añadir un difunto** a un nicho ya ocupado (`DifuntoAsignarController`, `WorkflowInhumacionController`) → todos los `inhumado` actuales pasan automáticamente a `restos`. El nuevo difunto es el `inhumado` activo con `es_titular = true`.
- Al pulsar **”Exhumar”** → el difunto pasa a `restos` (conserva `sepultura_id`), el nicho queda `libre`.
- El nicho se libera cuando **no quedan difuntos con `estado_inhumacion = 'inhumado'`** (los restos no bloquean).
- `CemnSepultura::difuntoTitular()` filtra `estado_inhumacion = 'inhumado'` para devolver solo el activo.

### Cambios en SepulturaDetalleController

Movimientos: cargados ahora para **todo el nicho** (por `sepultura_origen_id` o `sepultura_destino_id`), con nombre del difunto en cada registro. Ya no dependen de si el titular activo sigue en el nicho.

### Cambios en SepulturaInfoPanel.vue

# Devlog — 2026-05-11 (sesión tarde) — Cuadrícula reactiva + ExhumacionModal simplificado

## 1) Cuadrícula de nichos reactiva (verde ↔ rojo al instante)

**Problema**: al exhumar o añadir un difunto, el panel de sepultura se actualizaba pero la cuadrícula de nichos del bloque mantenía el color antiguo hasta que se cerraba y reabría.

**Solución**: `SepulturaInfoPanel` ahora emite `@changed { id, estado, nombre }` al final de cada `load()`. Los componentes que contienen la cuadrícula escuchan este evento y actualizan la celda local sin recargar toda la grid.

Ficheros modificados:
- `SepulturaInfoPanel.vue` — `defineEmits` añade `'changed'`; emit en `load()` tras `emit('loaded', ...)`.
- `BloqueGridView.vue` — `@changed=”onSepChanged”` en `SepulturaInfoPanel`; función `onSepChanged` actualiza `sepulturas.value[i].estado` y `tooltip_nombre`. CSS: `transition: background-color 400ms ease`.
- `BloqueGridModal.vue` — ídem.
- `DashboardPage.vue` — `@changed=”onSepChanged”` actualiza `allSepulturas` (mapa Leaflet).
- `CrudSepulturas.vue` — `@changed=”onSepChanged”` actualiza `items[i].estado` (columna Estado tabla).

## 2) ExhumacionModal simplificado

El modal anterior ofrecía elegir entre “restos” y “exhumado completo”, lo que generaba confusión. Rediseñado con comportamiento único:

- Siempre envía `estado_resultado = 'restos'` → la persona queda vinculada al nicho como restos físicos.
- El nicho pasa a `libre` porque ya no hay `inhumado` activos.
- Documento de sanidad es **opcional** (antes era obligatorio y bloqueaba el flujo).
- Mensaje explicativo claro en el modal: “La persona quedará como restos · el nicho quedará libre”.
- El modal cierra primero y luego emite `@exhumado`, que dispara `load(props.sepulturaId)` en el panel → recarga instantánea.

## 3) Corrección bug `onExhumado`

`onExhumado()` llamaba a `cargar()` que no existía → el panel no se recargaba tras la exhumación. Corregido a `load(props.sepulturaId)`.

## 4) Historial de movimientos del nicho

`SepulturaDetalleController` ahora carga movimientos por `sepultura_origen_id OR sepultura_destino_id` en lugar de solo los del `difuntoTitular`. Así el historial se mantiene aunque el titular ya no esté en el nicho (exhumado). Cada entrada del historial incluye `difunto_nombre` para identificar quién generó el movimiento.

## 2) Sistema de usuarios

### Backend — `UsersAdminController`

Nuevo controlador en `app/Http/Controllers/Admin/UsersAdminController.php`:
- `index()` — devuelve `{ users, roles, permissions }` (lista completa + catálogos Spatie).
- `store()` — crea usuario + asigna roles iniciales. Valida unicidad de `username` y `email`.
- `update()` — edita nombre, email, username, contraseña (nullable → no cambia si vacío).
- `destroy()` — no permite borrar la propia cuenta del usuario autenticado.
- `updateRoles()` / `updatePermissions()` — `syncRoles` / `syncPermissions` de Spatie.
- Todos los métodos requieren permiso `cementerio.admin`.

- Modal "Roles y permisos": checkboxes de roles y permisos directos cargados desde el backend.
- Confirmación de borrado con guard "no puedes borrarte a ti mismo".
- Solo visible en sidebar si `auth.hasPermission('cementerio.admin')`.

### Auth store — helpers añadidos

`resources/js/stores/auth.js`:
